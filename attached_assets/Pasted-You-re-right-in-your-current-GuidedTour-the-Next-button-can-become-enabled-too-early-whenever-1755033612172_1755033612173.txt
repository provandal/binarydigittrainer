You’re right: in your current `GuidedTour` the **Next** button can become enabled too early whenever a step has a `validation` function but `waitForAction` is not `true`. Your component defaults to `validationPassed = true` for any step that doesn’t explicitly set `waitForAction`, even if a `validation` exists. That’s why Step 10 can “unlock” early.

Below is a safe fix that makes the tour **require validation whenever a step defines either `validation` OR `waitForAction`**. It also lets you “mark complete” from the parent even if a step has no `validation` function (useful for one-click actions).

## How to define Step 10 in the parent

Make sure Step 10 **requires** validation (either set `waitForAction: true` or provide a `validation` function — I recommend both):

```ts
// In the parent where you define tourSteps:
{
  id: "waitAllTraining",
  title: "Training the full plan",
  content: `
    <p>We’ll advance when <strong>all samples across all epochs</strong> finish.</p>
    <p>Progress: ${trainedSampleCount} / ${totalPlannedSamples}</p>
  `,
  target: ".automated-training-controls",
  pin: "bottom-left",
  waitForAction: true,            // ensures GuidedTour treats this as gated
  autoAdvanceOnValid: false,      // optional: keep it manual (enable Next when done)
  validation: () => {
    const allPlanned = (trainingExamples.length || 0) * (numberOfEpochs || 0);
    return !isAutoTraining && trainedSampleCountRef.current >= allPlanned;
  },
}
```

And from the parent, **ask the tour to re-check** whenever progress changes:

```ts
// Hold the trigger from GuidedTour
const tourTriggerValidationRef = useRef<null | (() => void)>(null);

<GuidedTour
  isOpen={isTourOpen}
  onClose={() => setIsTourOpen(false)}
  onReset={resetNetwork}
  tourSteps={tourSteps}
  onValidationTrigger={(trigger) => { tourTriggerValidationRef.current = trigger; }}
/>

useEffect(() => {
  tourTriggerValidationRef.current?.();
}, [trainedSampleCount, trainingCompleted, isAutoTraining, numberOfEpochs, trainingExamples.length]);
```

> Important: only set `trainingCompleted = true` **once all samples in all epochs** are processed. Don’t set it in a per-sample loop.

---

### Why this fixes your case

* Previously, a step that **had a `validation`** but didn’t set `waitForAction` got `validationPassed = true` by default (due to the “no wait required” branch), enabling **Next** prematurely.
* Now, **any step with `validation` or `waitForAction` is considered gated**, starts as “not passed,” and only unlocks when `triggerValidation()` sets it true or `validation()` returns true.
* Step 10 won’t enable **Next** until your dataset×epochs loop has fully completed.
